Git Flow

Git Flow è una metodologia di branching pensata per gestire in modo ordinato lo sviluppo, le release e le correzioni in un progetto. È particolarmente utile nei team o nei progetti complessi.

Git Flow si basa su due branch permanenti:

  1. master: contiene il codice sempre stabile e rilasciato. Ogni commit su master rappresenta una versione pronta per la produzione.

  2. develop: è il ramo in cui avviene lo sviluppo quotidiano. Una volta che develop è stabile e completo, viene fuso in master per creare una release.


A questi si aggiungono branch temporanei:

  1. Feature branches

    2. Nome tipico: feature/nome-funzionalità
    Scopo: sviluppare nuove funzionalità.
    Partenza: da develop.
    Ritorno: merge in develop.


  2. Release branches

    3. Nome tipico: release/1.2.0
    Scopo: preparare una release (bugfix, version bump, metadati).
    Partenza: da develop quando è pronta una versione
    Ritorno: merge in master e merge in develop.


  3. Hotfix branches

    4. Nome tipico: hotfix/1.2.1
    Scopo: correggere bug urgenti in produzione.
    Partenza: da master.
    Ritorno: merge in master e merge in develop.

Origin

Origin è semplicemente il nome del remoto principale. Quando cloni una repository con:
git clone
Git crea automaticamente un remoto chiamato origin, che punta alla URL da cui hai clonato.
Serve per abbreviare i comandi:
invece di git push https://github.com/...
scrivi git push origin.

Cos’è origin/master o origin/branch?

E' un puntatore in sola lettura che Git mantiene per ricordarsi dove si trovava quel branch sul remoto l’ultima volta che hai fatto fetch/pull.
Rappresenta lo stato del branch remoto al momento dell’ultimo aggiornamento.

Tu fai git pull
Git aggiorna origin/master per farlo coincidere con il vero master su GitHub.

Tu fai nuovi commit in locale
master avanza, ma origin/master resta fermo finché non fai git fetch o git push.

Caso particolare: origin/HEAD

È solo un alias, un puntatore simbolico che indica il branch predefinito del repository remoto.
Di solito origin/HEAD → origin/master.
Serve a Git per sapere quale branch considerare come “default” quando cloni un repository.

A. Crei feature branch partendo da develop

git checkout develop
git pull
git branch feature/xyz
git checkout feature/xyz

E poi lavori come un negro... (vedi B)

Nota: quando faccio checkout develop e poi git pull lui in automatico esegue:
git pull origin develop
Lo stesso discorso vale per il master.

Perché succede? Ogni branch locale ha un suo “upstream branch”, cioè il branch remoto a cui è collegato (in questo caso origin/master o origin/develop).
Lo vedi con:
git branch -vv

Come collegare il branch locale al branch remoto: git branch --set-upstream-to=origin/nome-branch

B. Lavori grossi (User Story)

Una volta che hai creato il tuo branch e ti sei spostato su di esso (git checkout feature/xyz), puoi iniziare a lavorare.
  • Aggiungere file: git add file1 file2 ... filen

  • Fare commit regolarmente per salvare: git commit -m "Descrizione delle modifiche"

  • Quando una funzionalità è pronta, fai il merge su develop: git merge develop

  • Ora sei pronto per pushare il tuo develop (ricordati che stai lasciando indietro il master apposta come versione stabile): git push origin develop

C. Lavori piccoli

Se si seguiamo la tecnica Gitflow ma la modifica è minima da non far sborrare i professori, è inutile creare un branch apposta, puoi invece modificare il develop portando avanti quel branch.
  • git checkout develop

  • git add file1 file2 … filen

  • git commit -m "descrizione-commit"

  • git push

Ricordati che in questo modo stai lavorando esclusivamente su develop. Quando esegui git push, stai implicitamente dicendo git push origin develop. Se invece vuoi aggiornare anche il master perché sei sicuro che la versione in develop sia stabile, segui la procedura D.

D. Quando vuoi fare una release

Una volta che develop è stabile, quindi dopo aver pushato il develop sul remoto nel caso in cui hai tu la versione più aggiornata di develop oppure pullando il develop aul remoto nel caso in cui la tua versione di develop non è aggiornata:

Vai su master

  • git checkout master

  • git pull

  • git merge develop

  • git push origin master

Nota per Sam: tenere sincronizzati master e develop

Quando master avanza (es. hotfix urgente), porti la modifica su develop:

Vai su master

  • git checkout develop

  • git pull

  • git merge master

  • git push origin develop

Eliminare branch in locale

git branch -d nome-branch

Eliminare branch sul remoto

git push origin --delete nome-branch

Eliminare commit selettivamente

git rebase -i HEAD~5
  • pick → tenere il commit

  • drop → eliminarlo

  • edit → modificarlo

  • squash → unirlo al commit precedente

Spostare un branch su un altro commit e aggiornare la storia locale

git reset --hard hash-commit: Sposta il branch attuale su quel commit. Tutti i commit che non avranno più riferimenti verranno rimossi automaticamente da Git (garbage collection).

Per aggiornare la storia anche sul remoto

git push --force: Aggiorna solo il branch remoto dove sei. Se vuoi aggiornare anche l'altro branch ripeti le stesse operazioni.
