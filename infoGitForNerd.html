Origin

Origin è semplicemente il nome del remoto principale. Quando cloni una repository con:
git clone
Git crea automaticamente un remoto chiamato origin, che punta alla URL da cui hai clonato.
Serve per abbreviare i comandi:
invece di git push https://github.com/...
scrivi git push origin.

Cos’è origin/main o origin/branch

E' un puntatore in sola lettura che Git mantiene per ricordarsi dove si trovava quel branch sul remoto l’ultima volta che hai fatto fetch/pull.
Rappresenta lo stato del branch remoto al momento dell’ultimo aggiornamento.

Tu fai git pull
Git aggiorna origin/main per farlo coincidere con il vero main su GitHub.

Tu fai nuovi commit in locale
Main avanza, ma origin/main resta fermo finché non fai git fetch o git push.

Caso particolare: origin/HEAD

È solo un alias, un puntatore simbolico che indica il branch predefinito del repository remoto.
Di solito origin/HEAD → origin/main.
Serve a Git per sapere quale branch considerare come “default” quando cloni un repository.

A. Crei feature branch partendo da develop

git checkout develop
git pull
git branch feature/xyz
git checkout feature/xyz

E poi lavori come un negro... (vedi B)

Nota: quando faccio checkout develop e poi git pull lui in automatico esegue:
git pull origin develop
Lo stesso discorso vale per il main.

Perché succede? Ogni branch locale ha un suo “upstream branch”, cioè il branch remoto a cui è collegato (in questo caso origin/main o origin/develop).
Lo vedi con:
git branch -vv

Come collegare il branch locale al branch remoto: git branch --set-upstream-to=origin/nome-branch

B. Lavorare come SamuelBravi

Una volta che hai creato il tuo branch e ti sei spostato su di esso (git checkout feature/xyz), puoi iniziare a lavorare.
  • Aggiungere file: git add file1 file2 ... filen

  • Fare commit regolarmente per salvare: git commit -m "Descrizione delle modifiche"

  • Quando una funzionalità è pronta, fai il merge su develop: git merge develop

  • Ora sei pronto per pushare il tuo develop (ricordati che stai lasciando indietro il main apposta come versione stabile): git push origin develop

C. Quando vuoi fare una release

Una volta che develop è stabile (quindi dopo aver pushato il develop sul remoto):

Vai su master

  • git checkout master

  • git pull

  • git merge develop

  • git push origin main

Nota per Sam: tenere sincronizzati main e develop

Quando main avanza (es. hotfix urgente), porti la modifica su develop:

Vai su master

  • git checkout develop

  • git pull

  • git merge main

  • git push origin develop
